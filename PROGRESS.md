## Decisões da noite

**2026-07-05 (issue 018 — UI histórico de fornadas)**

1. **GRÁFICO SEM LIB — SVG próprio, não Chart.js/uPlot**: Trendline de margem/desperdício renderizada com `<svg>` puro em trendChart.ts (~80 linhas HTML+JS). Chart.js (~254kB minificado) ou uPlot (~30kB) reconstruiriam visual que mockup + design system já entregam via classes CSS. Motivo: mockup é SVG, design-system.css exporta todas as classes de estilo; custo de lib vs reuso visual existente desfavorável. Documentado como helper `svg()` em dom.ts (createElement para SVG namespace). Regra 1: libs para não-trivial; gráfico neste escopo é plotagem direta de pontos+linha (trivial).

2. **parseLocalDate como dono único de parse de data local — nunca new Date('aaaa-mm-dd')**: Novo função em format.ts que lê string 'aaaa-mm-dd' e cria Date meia-noite LOCAL (sem UTC shift). Pattern: `new Date(year, monthIndex, day)` nativo JS garante hora local 00:00:00. Usado exclusivamente por historyView e bakeForm para entrada/exibição. Bloqueador de bug 021: UI nunca faz `new Date('2026-07-05')` que cria meia-noite UTC e exibe deslocada ao usuário. Decisão: centralizar em format.ts, exportar, usar ubiquamente.

3. **Tabela lista TODAS as fornadas do filtro (planejadas com badge), mas KPIs/gráfico respeitam intervalo**: historyView renderiza fornadas confirme + planejadas (`isPlanned=true`) com badge visual "planejada" (laranja/design-system). KPIs (desperdício, margem, vendidas) PRÉ-filtram `!isPlanned` ANTES de groupBy (mitiga bug 021 UI calculando agregados com dados futuros). Gráfico (linha tempo) idem — eixo X é dias/semanas/meses, eixo Y métrica, mas só pontos de fornadas **confirmadas** aparecem. Planejadas adicionam badges nas datas, não pontos gráfico. Motivo: usuário vê contexto completo (incluindo planejadas), mas métricas/comparação são fatos (confirmadas apenas).

4. **Edição inline produzida/vendida (colunas do mockup), custo/data/receita fora**: historyView renderiza coluna "Produzidas" e "Vendidas" como `<input type="number">` editáveis (input→parseDecimal→store.update(BakeEntry)). Colunas "Data", "Receita", "Custo Unitário" são readonly (derivadas ou imutáveis após confirmação). Custo da produção (snapshot §14.2) é imutável — se cliente quer ajuste, nova fornada ou feature 019+. Mockup exibe 4 colunas metadados (Data/Receita/Custo) + 2 ações (Confirmar/Excluir) + 2 editáveis (Produzidas/Vendidas) — implementado exatamente isso. Simplificação: excluir (não editar data/receita, apenas trash).

5. **Snapshot do form BakeEntry gravado em 2 casas via moneyPlain — correto §14.2 (novo estado puro)**: Quando usuário clica "Confirmar planejada" em bakeForm, `totalCost`/`unitCost`/`unitSalePrice` são snapshots da época (via `moneyPlain` formatter — 2 casas sem símbolo R$, armazenam número puro). `cost: 8.86, unitCost: 4.43, unitSalePrice: 7.38` (exemplo). Snapshot nunca muda retroativamente se receita muda. Motivo: BakeEntry é **novo estado puro** (§14.2 "fixa fatos da fornada em um instante"), não referência a Recipe. Não viola "storage persiste estado cru" (decisão 011) porque BakeEntry é entidade própria, não derivada. Revisor: confirmar aceitável.

**2026-07-05 (issue 017 — Minhas Receitas + backup da tela)**

1. **Auto-save persiste Recipe pós-recalculate (derivados preenchidos) — adiado para decisão cliente**: Na tela Calculadora, ao carregar uma receita via `?recipe=<id>`, o estado exibido passa por `recalculate` automaticamente (core §1.6). Ao digitar qualquer campo, `store.update` clona a receita, aplica mutação, roda recalc e persiste via `recipeStore.update` (localStorage). Conseguência: Recipe no storage contém ingredientes com `currentWeight`/`currentPercent` DERIVADOS preenchidos (ex.: abrir receita em modo %→peso, Água 700g preenchida). Isoladamente é inócuo (recalculate idempotente re-deriva ao abrir), mas desvia do "storage guarda estado cru" da decisão 011.3 (estado COMPLETO mas SEM derivados). Revisor humano: confirmar se é aceitável, ou restaurar pattern: storage persiste SÓ fields editáveis (% ou peso conforme modo), re-deriva na carga.

2. **"Criar em branco" (§2.F) adiado — mockup tem 1 botão, implementação usa goldenSeed**: O mockup `receitas.html` desenha um único botão "+ Nova receita" que presumivelmente abre diálogo com campos em branco. Implementação issue 017 reutiliza `recipeStore.create(null)` que alimenta goldenSeed (valores padrão §12 — 1000g Farinha etc). Bloqueador zero (receita funcional nasce imediatamente); débito: adicionar diálogo/form "criar em branco" explícito que suba do zero.

3. **Confirm/prompt nativos injetáveis em vez de modal — design system não tem modal, zero dep**: Excluir receita (`window.confirm` "Excluir ... Fornadas órfãs ficarão..."`), renomear (`window.prompt("Novo nome")`), restaurar backup fail (`alert` pt-BR). APIs nativas bloqueiam thread — less UX-friendly que modal, MAS design-system.css não exporta componente modal, seria nova feature design 022+. Padrão: funções recebem `deps` injetáveis (`confirm`, `prompt`, `alert` sobrescritos em testes jsdom com vi.fn()). Produção usa native. Revisão: cliente pode decidir upgrade para modal se achar UX critical.

4. **Abrir receita aponta para index.html, não calculadora.html**: O mockup citava arquivo "calculadora.html" (nome de exemplo). Implementação navega para `index.html?recipe=<id>` (arquivo real da tela Calculadora, shell único). Link funciona offline (App é MPA Vite puro). Precisão: "Abrir" é `<a href="/index.html?recipe=${id}">Abrir</a>` em vez de rota/história/router-lib (zero deps, regra de ouro 1).

**2026-07-05 (issue 013 — histórico de fornadas)**

1. **wastageRate/averageProfitMargin do período = agregado ponderado, não média aritmética das % por fornada**: spec §14.4 diz "média", leitura defensável. Solução implementada: wastageRate = totalWastage/totalProduced × 100 (taxa global do período), averageProfitMargin = totalProfit/totalRevenue × 100 (margem global do período). Motivo: sincronização com BakeEntry.totalCost (§14.3 é snapshot do tempo de confirmação) — somar custo/receita de período requer ponderar por volume, não média direta das %/fornada. Decisão adotada no código; revisor humano confirmar se alinha à intenção.

2. **Datas de agrupamento 100% locais via formatDate — UI 017/018 NÃO deve gravar date como ISO UTC meia-noite**: agrupador genérico groupBy usa formatDate (getters locais, sem toISOString). Chaves lexicográficas aaaa-mm-dd são strings comparáveis (§14.5 levanta "comparação métrica-a-métrica"). Sem date-fns — operações triviais (~8 linhas) em Date puro (new Date(y, mIndex, d) = meia-noite local, getDay() nativo). MDN consultada. Bloqueio de implementação: UI 017/018 deve receber `date` local do padeiro, nunca UTC deslocado. Quando entregar BakeEntry, usar a data do input sem conversão.

3. **Sem date-fns — agrupamento trivial ~8 linhas, MDN getDay citada**: mondayOf, groupBy, groupByDay/Week/Month são ~40 linhas puras, sem lib. `date-fns` seria nova devDependency para lógica trivial. Motivo: regra de ouro 1/2 (libs consolidadas para não-trivial; reuso total). A aritmética de Date aqui não é complexa — getDay()%7 offset, new Date(y, m±1, d) para bordas. Link MDN documentado no cabeçalho do arquivo.
